Cohort System — Page Structure

How should the cohort system be organized? The architect needs to create/edit cohorts. The business user needs to see population risk at a glance.

A

Single Page — List + Detail

One "Cohortisation" page. Top section: cohort cards (like pathway list). Click a cohort to expand into detail view showing criteria, scoring engine, population stats. Business users see the same page — the population dashboard is part of each cohort's detail.

/dashboard/cohortisation
  ├─ Cohort list (cards: name, condition, patient count, tier breakdown mini-chart)
  ├─ [+ Create Cohort] button
  └─ Click card → /dashboard/cohortisation/:id
      ├─ Tab: Population (distribution chart, risk pool stats, patient table)
      ├─ Tab: Criteria (eligibility rules editor)
      ├─ Tab: Scoring Engine (optional — component builder, weights, tiers)
      └─ Tab: Linked Pathways (which pathways reference this cohort)

Pros

  • Everything in one place
  • Natural drill-down from list → detail
  • Mirrors the pathway list → editor pattern

Cons

  • Business user has to know which cohort to click
  • No cross-cohort population overview
B

Split — Population Dashboard + Cohort Builder

Two sub-pages under Cohortisation. The default view is a population dashboard (all cohorts, cross-cohort stats, risk pool overview). A separate "Cohort Builder" sub-page is where architects create/edit cohort definitions.

/dashboard/cohortisation (default: Population Dashboard)
  ├─ Cross-cohort population overview (total members, unassigned, tier distribution per cohort)
  ├─ Cohort summary cards (name, member count, avg CRS, trending arrows)
  ├─ Risk pool table (filterable by cohort, tier, condition)
  └─ [Manage Cohorts →] link

/dashboard/cohortisation/builder (Cohort Builder)
  ├─ Cohort list + [Create Cohort]
  └─ /dashboard/cohortisation/builder/:id
      ├─ Criteria editor
      ├─ Scoring engine (optional)
      └─ Preview (run against population, see who matches)

Pros

  • Business user lands on dashboard — sees risk pools immediately
  • Builder is separate, focused workspace for architects
  • Cross-cohort view answers "how is our population distributed?"

Cons

  • Two sub-pages to maintain
  • Navigation between dashboard and builder could feel disconnected
C

Dashboard-First with Inline Builder

Population dashboard is the landing page. Each cohort card is expandable — click to see detail + edit criteria inline. Creating a new cohort opens a slide-over panel or dialog, not a new page. Keeps everything in one flow.

/dashboard/cohortisation
  ├─ Population overview bar (total members, assigned %, unassigned count)
  ├─ Cohort cards grid (each card: name, condition, member count, mini tier chart)
  │   └─ Click card → expands inline: criteria summary, scoring summary, population table
  │       └─ [Edit Cohort] → sheet/drawer with full criteria + scoring editor
  ├─ [+ Create Cohort] → sheet/drawer with builder
  └─ Risk pool table (all members, filterable by cohort/tier)

Pros

  • Single page, no navigation
  • Dashboard-first for business users
  • Editing is contextual (drawer), not a page switch

Cons

  • Drawer may be too cramped for complex scoring engine config
  • Inline expansion can get cluttered with many cohorts
